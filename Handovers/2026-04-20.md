### 4. Contradiction detector — critical fix (false positive)

**The problem:** A false positive was caught before the pack was published. The keyword "frugal" matched the sentence *"The app needs to be frugal in complexity costs"* — a technical sentence, not a self-claim. The pack had reported: *"John stated he is frugal/careful with money"* — entirely fabricated.

This was identified as a configuration error producing a false signal, not a weak signal requiring more data. Rule established:

> **Any configuration error producing a false signal must be fixed immediately. False signal ≠ weak signal. Weak signals require more data. False signals undermine the forensic trustworthiness of the entire tool.**

**Fix 1 — Verbatim quote storage:**
`_find_self_claim_sentence()` now returns the exact triggering sentence instead of `True/False`. Every contradiction signal now stores:
- `evidence` field: `Verbatim claim: "<exact sentence>"`
- `metadata["verbatim"]`: the quote for downstream use

Every signal is now self-auditable — the claim can always be verified without re-reading source data.

**Fix 2 — LLM-based claim extraction:**
`_extract_claims_llm()` added. When an API key is available (always in production):
1. Pre-filters messages down to candidates containing any claim keyword (reduces token cost)
2. Sends up to 200 candidates to Claude Haiku
3. Haiku extracts only genuine first-person identity claims with strict criteria — excludes technical uses, third-party mentions, aspirational statements
4. Returns structured list: `{"claim_type": str, "verbatim": str}`

Keyword matching with first-person window check retained as fallback for tests and offline use.

**Post-fix result:** 0 false positives on John's full dataset. The two contradictions in v5 pack were confirmed genuine by re-running with the LLM path.

## Key design decisions made this session

1. **False signal vs weak signal distinction** — this is now a standing principle. Any signal that fires incorrectly due to configuration is a false signal and must be fixed before any run produces output. Weak signals (real pattern, insufficient data) are acceptable — they get hedged in the pack.

2. **LLM for claim extraction, not transaction categorisation** — the LLM path is used only where natural language understanding is required (identifying self-claims). Transaction categorisation stays as keyword matching (appropriate for structured data).

3. **Public repo** — decided in favour of public on the grounds that: execution is the moat, not the code; the idea of individual data sovereignty is worth staking publicly; timestamped commits are the closest thing to a patent in open source.
