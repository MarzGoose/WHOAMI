Ship strategy — three paths

These aren't mutually exclusive. The question is what comes first and what that unlocks.

TIMELINE
Week 1–2: Python CLI working on your own data
Week 3–4: Claude exports + bank CSV working
Month 2: iMessage + browser + Electron shell
Month 3: Freemium launch, GitHub open core

A — Ship fast, iterate in public

Build the CLI first. Use it on your own data. Put it on GitHub. Post the results (not the code — the output) on TikTok/Instagram. Real signal from real data is the content. Desktop app comes after the CLI proves itself.

For

  • Working version in ~2 weeks
  • GitHub timestamp = prior art protection
  • Content generates itself (your own mirror)
  • Feedback shapes the desktop app

Against

  • CLI limits early audience to technical users
TIMELINE
Week 1–3: Landing page + concept video
Week 4–6: TikTok/IG build-in-public content
Month 2–3: Waitlist grows while you build
Month 4: Launch to warm audience

B — Hype first, build behind the curtain

Build the landing page and concept video now. Start the waitlist. Post the idea and the problem — not the product. Build in private. Launch when it's ready to an audience already waiting.

For

  • Demand validated before full build
  • Warm audience on launch day
  • Protects implementation details

Against

  • Longer until you get real feedback
  • Concept can be copied before you ship
  • Hype without product can deflate

C — Open core ★ recommended

CLI engine on GitHub (open source, prior art, developer credibility). Desktop app is the paid product. GitHub is the marketing to developers; TikTok/IG reaches everyone else. Both audiences, two channels.

For

  • Prior art protection from day one
  • Developers find it, non-devs buy the app
  • Community improves parsers for free
  • GitHub stars = social proof

Against

  • Needs clear boundary between free/paid

On GitHub: yes, absolutely — open-source CLI core establishes prior art and builds developer credibility. The desktop app stays proprietary.
